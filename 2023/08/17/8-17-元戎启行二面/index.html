操作系统中的锁

  • 自旋锁 spinlock
    自旋锁的主要特征是使用者在想要获得临界区执行权限时，如果临界区已经被加锁，那么自旋锁并不会阻塞睡眠，等待系统来主动唤醒，而是原地忙轮询资源是否被释放加锁。自旋锁的主要特征是使用者在想要获得临界区执行权限时，如果临界区已经被加锁，那么自旋锁并不会阻塞睡眠，等待系统来主动唤醒，而是原地忙轮询资源是否被释放加锁
  • 互斥锁 mutex
    使用者使用互斥锁时在访问共享资源之前对进行加锁操作，在访问完成之后进行解锁操作，谁加锁谁释放，其他使用者没有释放权限。 加锁后，任何其他试图再次加锁的线程会被阻塞，直到当前进程解锁，系统将阻塞的线程进行唤醒。
  • 读写锁 rwlock
    读写锁也叫共享互斥锁：读模式共享和写模式互斥。读写锁有读加锁状态、写加锁状态和不加锁状态三种状态，当读写锁在写加锁模式下，任何试图对这个锁进行加锁的线程都会被阻塞，直到写进程对其解锁。
    读优先的读写锁：读写锁rwlock默认的也是读优先，也就是:当读写锁在读加锁模式先，任何线程都可以对其进行读加锁操作，但是所有试图进行写加锁操作的线程都会被阻塞，直到所有的读线程都解锁，因此读写锁很适合读次数远远大于写的情况。这种情况需要考虑写饥饿问题，也就是大量的读一直轮不到写，因此需要设置公平的读写策略。
  • RCU锁
    RCU锁是读写锁的扩展版本，简单来说就是支持多读多写同时加锁。RCU锁翻译为Read Copy Update Lock，读-拷贝-更新 锁。
    Copy：写者在访问临界区时，写者将先拷贝一个临界区副本，然后对副本进行修改；
    Update：RCU机制将在在适当时机使用一个回调函数把指向原来临界区的指针重新指向新的被修改的临界区，锁机制中的垃圾收集器负责回调函数的调用。
    更新时机：没有CPU再去操作这段被RCU保护的临界区后，这段临界区即可回收了，此时回调函数即被调用。
    写拷贝开销较大，适合读多写少（路由表的查询更新、设备状态表更新）。
  • 条件变量 condition variables
    条件变量本身不是锁，条件变量可以主动等待和唤醒，通常和互斥锁配合使用。可以实现基于业务逻辑，将阻塞的互斥锁进行异步唤醒。
  • 可重入锁和不可重入锁(递归锁和非递归锁)
    两个函数都需要加同一把锁，而这两个函数之间有调用关系。如果锁是不可重入的，那么执行到被调用的函数加锁处时就会死锁。而可重入锁不会，允许递归获取同一个锁。
    Windows中的Mutex和Critical Section是可递归的，Linux下的pthread_mutex_t默认是非递归的，可以显示设置PTHREAD_MUTEX_RECURSIVE属性更改为可递归的。

C++11中的锁

  1. std::mutex
    在头文件<mutex>里，可以上锁lock()和解锁unlock()，不过不建议这样做。可以用RAII的std::lock_guard()和std::unique_lock()来自动管理std::mutex的上锁和解锁。底层封装pthread_mutex_t。

  2. std::timed_mutex
    额外提供了try_lock_for()和try_lock_until()

  3. std::recursive_mutex
    可重入锁，递归锁。底层实现好像和std::mutex一样？应该因为Linux底层的pthread_mutex_t的类型属性被设置为PTHREAD_MUTEX_RECURSIVE，已经是可重入锁了，所以是一样的。

  4. std::recursive_timed_mutex
    额外提供超时功能。

  5. std::shared_mutex c++17
    共享锁，允许以共享模式加互斥锁。额外有lock_shared()和try_lock_shared()。当有线程获得独占模式的锁时其他线程都不能获得锁，当有线程获得共享模式的锁时，其他线程不能获得独占锁但能获得共享锁。

  6. std::condition_variable
    在头文件<condition_variable>里，配合互斥锁使用，实现同步机制。可以实现线程等待某一事件发生，才会继续执行。

    如果想让线程等待一次性事件，使用std::future。代表未来事件，可以选择转换运行模式。

    第一个参数可以指定运行模式，std::launch::async表示必须另外开启新线程执行任务函数，通过get()获取返回值。std::launch::deferred指定在当前线程上延后调用，通过wait()进行调用，同样用get()获取返回值。

通用互斥管理

  1. std::lock_guard c++11
    提供RAII机制，创建时加锁（等效调用lock()），销毁时解锁（等效调用unlock()）。不可移动，不可复制。

  2. std::unique_lock c++11
    std::lock_guard拥有的功能它都有。额外提供更多的灵活性，可以自己选择关不关联互斥量以及何时加锁何时解锁。可移动，不可复制。移动时，转换锁的所有权。

  3. std::shared_lock c++14
    创建时执行lock_shared()，销毁时执行unlock_shared()。且提供足够的灵活性。可移动，不可复制。移动时，转换锁的所有权。

  4. std::scoped_lock c++17

    是std::lock_guard的多锁版本，构造函数接收可变数量的锁。构造时等效对所有锁加锁，析构时对所有锁解锁。
    底层实现是，对第一个锁用unique_lock上锁，如果成功就不断递归地用unique_lock获取下一个锁，只要有一个unique_guard调用owns_lock()返回false则直接release()。

策略模式和工厂模式

策略模式

在策略模式中一个类的行为或算法可以在运行时更改。定义了一系统算法或策略，并将算法封装在独立的类中，使得他们可以互相替换。可以在运行时根据需要选择不同的算法，而不用修改代码。

解决多种算法相似的情况下，使用if else带来的复杂和难以维护。

本质上就是定义一个基类和虚函数，每种不同的策略去继承这个基类，重写虚函数，然后定义一个包装类，封装基类，只要通过包装类创建不同策略对象，用包装类中封装的基类指向策略类，去执行相应算法。

工厂模式

工厂模式是将对象的实例化过程抽象成流水线生产，将具体的实例化过程封装在工厂类中，这样就可以将对象的创建与使用的代码分离，避免繁琐的创建暴露在业务代码中。因为有的对象在创建前需要一些准备工作，封装在一起模块化更高。

具体来说，所有”产品“都继承自一个基类作为流水线，设置一个工厂类，每个流水线设置一个生产的方法，生产该流水线上的”产品“，流水线根据传递的消息生产不同的”产品“。

抽象工厂模式

流水线太多的话，工厂难以管理，而且耦合性太高，不易维护。

将不同的流水线抽象出来成为一个工厂，即一个工厂只管理一个流水线，每个工厂生产不同类型的产品。需要有一个超级工厂，来生产工厂。超级工厂就是所谓的抽象工厂，他的产品是具体的工厂即一条流水线。
